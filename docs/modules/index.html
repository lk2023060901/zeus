模块文档

Zeus游戏框架采用模块化架构设计，每个模块都有明确的职责和清晰的接口。本文档详细介绍游戏框架各个模块的功能、API和使用方法。

架构概览

模块层次结构

游戏应用层 (Game Applications)
    ├── game_server/           # 游戏服务器程序
    ├── tests/                 # 游戏功能测试
    └── examples/             # 游戏示例程序
                ↓
游戏逻辑层 (Game Logic Layer)  
    ├── game_world/           # 游戏世界管理
    ├── player_manager/       # 玩家管理系统
    └── combat_system/        # 战斗系统
                ↓
框架服务层 (Framework Services)
    ├── network/              # 网络通信模块
    ├── database/             # 数据存储模块
    ├── common/spdlog/        # 日志模块
    └── config/               # 配置管理模块
                ↓
基础设施层 (Infrastructure)
    ├── spdlog-1.15.3         # 高性能日志库
    ├── nlohmann-json-3.12.0  # JSON数据处理
    └── boost-1.88.0          # 网络和系统库

模块依赖关系

依赖原则： 上层模块依赖下层模块，下层模块不依赖上层模块，同层模块之间保持最小依赖。
模块 依赖 功能
Tests common::spdlog 功能验证、性能测试、集成测试
common::spdlog spdlog, nlohmann/json 日志管理、配置解析、文件操作
spdlog 无 高性能日志记录
nlohmann/json 无 JSON数据处理

模块特性

common::spdlog 模块特性

特性 实现方式 配置选项
日志轮转 daily_file_sink_mt, hourly_file_sink_mt rotation_type: "daily"/"hourly"
多级别日志 spdlog::level::level_enum level: "trace"/"debug"/"info"/"warn"/"error"/"critical"
线程安全 multi-threaded sinks (*_mt) 内置支持，无需额外配置
控制台输出 stdout_color_sinks_mt console_output: true/false
文件数量限制 max_files参数 max_files: 数字

API概览

主要API接口

ZeusLogManager

class ZeusLogManager {
public:
    // 获取单例实例
    static ZeusLogManager& Instance();
    
    // 初始化日志系统
    bool Initialize(const std::string& config_file = "");
    
    // 获取Logger
    std::shared_ptr<::spdlog::logger> GetLogger(const std::string& name);
    
    // 重新加载配置
    bool ReloadConfig();
    
    // 清理资源
    void Shutdown();
};

使用示例

#include "common/spdlog/zeus_log_manager.h"

int main() {
    using namespace common::spdlog;
    
    // 初始化
    if (!ZeusLogManager::Instance().Initialize("log_config.json")) {
        return 1;
    }
    
    // 获取Logger
    auto logger = ZeusLogManager::Instance().GetLogger("app");
    
    // 记录日志
    logger->info("应用程序启动");
    logger->warn("这是警告信息");
    logger->error("错误信息: {}", error_msg);
    
    return 0;
}

Logger操作API

// 基本日志记录
logger->trace("跟踪信息");
logger->debug("调试信息");
logger->info("一般信息");
logger->warn("警告信息");
logger->error("错误信息");
logger->critical("严重错误");

// 格式化日志
logger->info("用户 {} 登录成功，时间: {}", username, timestamp);
logger->error("文件 {} 打开失败，错误码: {}", filename, error_code);

// 条件日志
if (logger->should_log(spdlog::level::debug)) {
    logger->debug("调试信息: {}", expensive_debug_info());
}

// 设置日志级别
logger->set_level(spdlog::level::warn);

// 刷新缓冲区
logger->flush();

性能特性

📊 性能指标

测试场景 吞吐量 延迟
单线程日志 ~500,000 logs/sec < 1μs
多线程日志 ~1,000,000 logs/sec < 2μs
异步日志 ~2,000,000 logs/sec < 0.5μs

⚡ 优化特性

  • 零拷贝 - 最小化内存分配
  • 预分配缓冲区 - 减少动态分配
  • 无锁队列 - 高效线程间通信
  • 批量写入 - 减少系统调用
  • 编译时优化 - 格式字符串优化

内存使用

  • 基础开销: ~50KB (静态库)
  • 每个Logger: ~2KB
  • 缓冲区: 可配置 (默认8KB)
  • 第三方库: Header-only集成

最佳实践

🎯 Logger命名

  • 使用有意义的名称: "database", "network"
  • 按模块划分: "auth", "billing"
  • 避免特殊字符，使用小写字母
  • 保持名称简短但具有描述性

📁 文件组织

  • 按日期分目录: logs/2025/01/app.log
  • 按模块分文件: logs/auth.log, logs/db.log
  • 使用统一的文件扩展名: .log
  • 设置合理的轮转策略

🔧 性能优化

  • 生产环境使用INFO级别
  • 避免在热路径记录DEBUG日志
  • 使用异步日志处理高并发
  • 合理设置缓冲区大小

🛡️ 错误处理

  • 检查初始化返回值
  • 处理配置文件错误
  • 监控磁盘空间

模块集成

CMake集成

在您的项目中集成Zeus模块：

# 添加Zeus子项目
add_subdirectory(third_party/zeus/src/common)

# 创建您的目标
add_executable(my_app main.cpp)

# 链接Zeus模块
target_link_libraries(my_app PRIVATE common_spdlog)

# 包含头文件
target_include_directories(my_app PRIVATE 
    third_party/zeus/include
)

命名空间使用

// 推荐方式 - 明确命名空间
#include "common/spdlog/zeus_log_manager.h"

int main() {
    // 显式使用命名空间
    common::spdlog::ZeusLogManager::Instance().Initialize();
    
    // 或在局部作用域使用
    using namespace common::spdlog;
    auto logger = ZeusLogManager::Instance().GetLogger("app");
    
    return 0;
}
